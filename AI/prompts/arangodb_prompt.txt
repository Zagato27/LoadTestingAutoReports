ЗАДАЧА: Комплексный отчет по метрикам ArangoDB  в ходе ступенчатого нагрузочного тестирования

ИСХОДНЫЕ ДАННЫЕ:
- Полные данные по HTTP-запросам (total_requests_total)
- Полные данные по времени выполнения AQL-запросов (95-й перцентиль)
- Все дополнительные метрики, представленные в таблицах

РЕКОМЕНДУЕМЫЕ SLA (по мировой практике):
- Время выполнения AQL-запросов:
  * Нормальное: P95 < 100 мс
  * Предупреждение: P95 = 100-200 мс
  * Критическое: P95 > 200 мс
- Доступность (uptime):
  * Целевая: 99.9% (не более 43 минут простоя в месяц)
  * Минимальная: 99.5% (не более 3.6 часов простоя в месяц)
- Пропускная способность:
  * Должна поддерживаться стабильная производительность при пиковых нагрузках
  * Минимум 100 IOPS для каждого тома в production
- Общая производительность:
  * Apdex ≥ 0.85 (85% запросов выполняются в пределах целевого времени)

ТРЕБОВАНИЯ К АНАЛИЗУ:
1. Проанализировать ВСЕ доступные метрики без исключения
2. По каждой метрике предоставить:
   - Минимальное значение
   - Максимальное значение
   - Среднее значение
   - Времена и условия пиковых значений
   - Закономерности изменения

3. Отдельно проанализировать:
   - Уровень запросов (RPS) к ArangoDB по всем pod/instance
   - 95-ю перцентиль времени выполнения AQL-запросов в полном объеме
   - Корреляцию между нагрузкой и временем ответа для каждого экземпляра
   - Соответствие показателей рекомендуемым SLA

СТРУКТУРА ОТВЕТА:
1. Объём запросов
   • Детальный анализ ВСЕХ метрик HTTP-запросов
   • Разбивка по pod/instance с указанием всех значений
   • Изменение во времени с указанием всех выявленных закономерностей
   • Соответствие фактических значений целевым SLA

2. Время выполнения AQL (95-й перцентиль)
   • Полный анализ динамики времени выполнения
   • Все зафиксированные пики с указанием точных значений
   • Стабильность показателей в различные периоды
   • Сравнение с целевыми значениями SLA

3. Корреляция нагрузки и латенси
   • Исчерпывающий анализ взаимосвязи между всеми метриками
   • Все случаи одновременного роста запросов и задержек
   • Полная информация о периодах восстановления
   • Соответствие целевым SLA в периоды пиковой нагрузки

4. Итоговые выводы
   • Детальный анализ по каждому pod/экземпляру
   • Конкретные рекомендации на основе всех проанализированных данных
   • Перечисление всех выявленных узких мест
   • Общая оценка соответствия SLA и рекомендации по улучшению

ВАЖНО:
1. НЕ ПРИДУМЫВАТЬ ДАННЫХ. Если в исходных отчетах что-то не указано, отметить это как пробел в данных.
2. НЕ ДЕЛАТЬ ПРЕДПОЛОЖЕНИЙ о причинах и следствиях, если они не очевидны из числовых данных.
3. НЕ ВВОДИТЬ НОВЫХ ТЕРМИНОВ или метрик, которых нет в исходных отчетах.
4. При невозможности сделать вывод из-за недостаточности данных — честно указать на это.
5. Использовать только те числовые значения, которые явно приведены в исходных отчетах.
6. Не пропускайте никакие значения из таблиц. Если какое-то значение кажется незначительным, всё равно включите его в анализ с обоснованием.


ФОРМАТ ОТВЕТА:
